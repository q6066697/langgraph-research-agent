# langgraph-research-agent

ReAct-агент на [LangGraph](https://github.com/langchain-ai/langgraph): ищет актуальную информацию в вебе через Tavily и отвечает на вопросы, приводя источники (URL) в конце ответа. Поддерживает переключение LLM-провайдера (OpenAI или Anthropic) через переменную окружения.

## Схема графа

```
                ┌───────┐
   entry ─────► │ agent │◄─────────────┐
                └───┬───┘               │
                    │                   │
          tools_condition               │
                    │                   │
        ┌───────────┴───────────┐       │
        │                       │       │
   есть tool_calls?      нет tool_calls │
        │                       │       │
        ▼                       ▼       │
    ┌───────┐                 ┌─────┐   │
    │ tools │─────────────────┘ END │   │
    └───┬───┘                 └─────┘   │
        │                               │
        └───────────────────────────────┘
```

Поток: `agent` решает, нужен ли поиск. Если модель запросила вызов инструмента — граф идёт в `tools`, результат возвращается в `agent` для формирования ответа. Если вызов инструмента не нужен — граф завершается в `END`.

## Как это работает

**State (`Annotated[list, add_messages]`)** — состояние графа хранит единственное поле `messages`. Редьюсер `add_messages` при каждом обновлении не перезаписывает список, а добавляет новые сообщения к уже накопленной истории — это и есть механизм памяти диалога между узлами.

**Узел `agent`** — подставляет системный промпт перед историей сообщений и вызывает LLM с привязанными инструментами (`bind_tools`). Модель либо сразу формулирует ответ, либо возвращает сообщение с `tool_calls`, если ей не хватает данных для факта.

**`ToolNode`** — готовый узел из `langgraph.prebuilt`, который автоматически исполняет любой инструмент, запрошенный моделью в `tool_calls`, и оборачивает результат в `ToolMessage`. Здесь единственный инструмент — `TavilySearch(max_results=3)`.

**Условное ребро (`tools_condition`)** — тоже готовая функция из `langgraph.prebuilt`: проверяет последнее сообщение от `agent` на наличие `tool_calls`. Если они есть — направляет выполнение в `tools`, иначе — в `END`. После `tools` граф всегда возвращается в `agent`, чтобы модель учла результат поиска.

## Стек

- Python 3.12
- [LangGraph](https://github.com/langchain-ai/langgraph)
- [langchain-tavily](https://pypi.org/project/langchain-tavily/) — веб-поиск через Tavily
- LLM-провайдер: OpenAI (`langchain-openai`) или Anthropic (`langchain-anthropic`) — выбирается через `.env`

## Установка

```bash
git clone <URL-этого-репозитория>
cd langgraph-research-agent

python3 -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate

pip install -r requirements.txt

cp .env.example .env
```

Затем впишите в `.env` свои ключи:

```
OPENAI_API_KEY=...
ANTHROPIC_API_KEY=...
TAVILY_API_KEY=...
PROVIDER=openai        # или anthropic
```

Запуск:

```bash
python agent.py
```

## Известные ограничения

- **Свежесть ответа зависит от выдачи Tavily.** Агент не проверяет дату публикации источников и не делает reranking результатов поиска — он берёт первые `max_results` совпадений как есть. Если Tavily вернёт устаревшую или неавторитетную страницу выше более свежей, агент может процитировать именно её.
- Без фильтра по дате модель может смешать в ответе актуальные и устаревшие факты, особенно по быстро меняющимся темам (версии ПО, цены, состав команд и т.п.) — стоит перепроверять важные ответы вручную.
- Поиск ограничен тремя результатами (`max_results=3`) — для сложных вопросов с противоречивыми источниками этого может быть недостаточно.
- Нет памяти между запусками процесса: история диалога живёт только в рамках одного сеанса `python agent.py`.
